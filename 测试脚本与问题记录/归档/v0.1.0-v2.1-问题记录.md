## 2026-04-30: 登录接口返回 500 Internal Server Error

### 问题描述
执行登录操作时，后端返回 HTTP 500 错误，前端控制台显示：
```
Failed to load resource: the server responded with a status of 500 (Internal Server Error)
```

API 响应：
```json
{
  "code": 500,
  "message": "服务器内部错误",
  "data": null,
  "timestamp": 1777531894599
}
```

### 根因分析

#### 1. bcrypt 模块缺失（前置问题）
首次排查发现 `bcrypt` 依赖未安装到 `node_modules` 中（虽然 `package.json` 中已声明依赖），导致 `auth.service.ts` 在 `import * as bcrypt from 'bcrypt'` 时加载失败。

**修复：** 重新执行 `npm install` 安装依赖。

> **注意：** 根目录下存在多余的 `package.json`（仅包含 `{"dependencies": {"bcrypt": "^6.0.0"}}`）和 `node_modules` 文件夹，干扰了 `backend/` 目录的 npm 命令执行。在删除根目录 `node_modules` 和 `package.json` 后，npm 又因找不到根 `package.json` 而报错。最终需要在根目录保留一个空的 `package.json`。

#### 2. JWT payload 与 options.expiresIn 冲突（核心原因）
修复 bcrypt 后重新测试，后端日志报出：
```
Error: Bad "options.expiresIn" option the payload already has an "exp" property.
```

**原因：** `jsonwebtoken` 库禁止同时在 payload 对象中设置 `exp` 字段 和 options 中设置 `expiresIn`。payload 中的 `exp: 0` 与模块配置的 `expiresIn: '7d'` 冲突，导致签名失败。

### 修复方案

#### 修改 1: `backend/src/modules/auth/auth.service.ts`
移除 payload 中手动设置的 `iat` 和 `exp` 字段，让 `JwtService.sign()` 根据 `JwtModule` 配置自动添加。

#### 修改 2: `backend/src/common/decorators/current-user.decorator.ts`
将 `JwtPayload` 接口中的 `iat` 和 `exp` 改为可选字段。

### 验证结果
```json
HTTP 200 OK
{
  "code": 200,
  "message": "登录成功",
  "data": {
    "user": { "id": 1, "username": "admin", ... },
    "accessToken": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
  }
}
```

### 经验教训
1. **JWT payload 不应手动设置 `iat`/`exp`**：`@nestjs/jwt` 的 `JwtModule` 配置中的 `signOptions.expiresIn` 会自动处理过期时间。
2. **根目录的 `package.json` 会影响子目录的 npm 行为**：如果根目录存在 `package.json`，npm 会将其视为 monorepo 工作区配置。
3. **依赖安装后需要验证**：`npm install` 成功后应检查关键依赖是否确实存在于 `node_modules` 中，特别是原生模块如 `bcrypt`。

## 2026-05-01: v0.2.0-p1 课程模块测试结果与问题记录

### 测试结果
- **测试时间：** 2026-05-01
- **测试范围：** 45 个测试用例（TC-C01 ~ TC-C44 + TC-C17b）
- **通过：** 39 个 | **失败：** 6 个 | **通过率：** 86.7%

### 失败用例分析

| 用例编号 | 用例名称 | 期望状态码 | 实际状态码 | 问题类型 |
|:--------:|----------|:----------:|:----------:|----------|
| TC-C21 | 学员无权创建课程 | 403 | 401 | 学员注册后 token 为空 |
| TC-C36 | 创建课程使用无效分类ID | 400 | 500 | 未校验分类是否存在 |
| TC-C37 | 学员越权更新课程 | 403 | 401 | 学员 token 为空 |
| TC-C40 | 创建章节时课程不存在 | 404 | 500 | 未校验课程是否存在 |
| TC-C41 | 创建课时时章节不存在 | 400 | 500 | 未校验章节是否存在 |
| TC-C43 | 学员访问管理端接口 | 403 | 401 | 学员 token 为空 |

### 根因分析与修复

#### 问题 1: 学员注册后 token 获取失败（影响 TC-C21/TC-C37/TC-C43）
**根因分析：** 注册接口返回的 `data` 中可能不包含 `accessToken`，或者注册时使用了已存在的用户名（唯一后缀冲突）。
**修复方案：** 测试脚本中注册学员后，使用注册的账号密码重新登录获取 token，确保 token 有效。

#### 问题 2: 创建课程时未校验分类是否存在（TC-C36）
**根因分析：** `courses.service.ts` 的 `create()` 方法直接调用 `this.courseRepository.save()`，没有先校验 `categoryId` 对应的分类是否存在。
**修复方案：** 在 `courses.service.ts` 的 `create()` 方法中注入 `CategoriesService` 或直接查询分类表，校验分类是否存在。

#### 问题 3: 创建章节时未校验课程是否存在（TC-C40）
**根因分析：** `chapters.service.ts` 的 `create()` 方法没有校验 `courseId` 对应的课程是否存在。
**修复方案：** 在 `chapters.service.ts` 的 `create()` 方法中注入 `CoursesService`，先校验课程是否存在。

#### 问题 4: 创建课时时未校验章节是否存在（TC-C41）
**根因分析：** `lessons.service.ts` 的 `create()` 方法没有校验 `chapterId` 对应的章节是否存在。
**修复方案：** 在 `lessons.service.ts` 的 `create()` 方法中校验章节是否存在。

#### 问题 5: 公开课程列表查询总数显示为 0（TC-C09/TC-C10/TC-C11）
**根因分析：** 测试脚本使用 `data.get('total',0)` 获取总数，但 API 返回格式为 `{ items: [...], meta: { total, ... } }`。
**修复方案：** 测试脚本中改为 `data.get('meta', {}).get('total', 0)` 获取总数。

#### 问题 6: 分类查询参数类型转换问题（TC-C10）
**根因分析：** `CourseQueryDto` 中的 `categoryId` 字段使用了 `@IsNumber()` 但缺少 `@Type(() => Number)` 转换器。
**修复方案：** 在 `CourseQueryDto` 的 `categoryId` 和 `teacherId` 字段上添加 `@Type(() => Number)` 装饰器。
