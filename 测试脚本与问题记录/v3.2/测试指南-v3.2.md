## 一、测试环境准备

### 1.1 前置条件

| 条件 | 说明 |
|------|------|
| MySQL | 已启动（docker compose up -d mysql） |
| Redis | 已启动（docker compose up -d redis） |
| MinIO | 已启动（docker compose up -d minio） |
| 种子数据 | 已执行 `node backend/scripts/seed.js && node backend/scripts/seed_courses.js` |
| 后端 | 已启动（npm run start:dev） |

### 1.2 支付宝配置说明

由于当前为开发/测试环境，`ALIPAY_ENABLED=false` 时支付宝功能处于禁用状态。在此状态下：
- `POST /payments/create` → 返回 400 "支付宝支付功能未配置，请联系管理员"
- `POST /payments/confirm` → 抛出 BadRequestException

如需完整测试支付流程，需在 `.env` 中配置有效的支付宝沙箱账号。

## 二、测试 API 清单

### 2.1 支付接口

| 方法 | 路径 | 角色 | 功能描述 |
|------|------|------|---------|
| POST | `/api/v1/payments/create` | student | 创建支付宝支付（生成支付链接） |
| POST | `/api/v1/payments/notify` | 无（公开） | 支付宝异步通知回调 |
| GET | `/api/v1/payments/query/:orderNo` | 任意登录 | 查询支付状态 |
| POST | `/api/v1/payments/confirm` | student | 手动确认支付（备用方案） |

### 2.2 订单接口（支付相关）

| 方法 | 路径 | 角色 | 功能描述 |
|------|------|------|---------|
| POST | `/api/v1/orders/create` | student | 创建订单 |
| POST | `/api/v1/orders/:id/cancel` | student | 取消订单 |
| POST | `/api/v1/admin/orders/:id/refund` | super_admin/operator | 管理端退款处理 |
| GET | `/api/v1/orders/check-purchase/:courseId` | 任意登录 | 检查课程购买状态 |

## 三、测试场景与用例

### 场景 1: 支付创建

**TC-PAY-01: 创建支付（正常流程）**
- 前置：学员登录 → 创建待支付订单
- 操作：调用 `POST /payments/create` 传入订单ID
- 预期：若支付宝未配置，返回 400 提示"支付宝支付功能未配置"
- 备注：生产环境配置后返回 `{ payUrl: "https://openapi.alipay.com/..." }`

**TC-PAY-02: 创建支付（订单不属于当前用户）**
- 前置：学员A创建订单 → 学员B尝试对该订单发起支付
- 操作：学员B调用 `POST /payments/create`
- 预期：返回 400 "无权操作此订单"

**TC-PAY-03: 创建支付（非待支付订单）**
- 前置：学员创建订单 → 取消订单
- 操作：对已取消订单调用 `POST /payments/create`
- 预期：返回 400 "仅待支付状态的订单可以发起支付"

**TC-PAY-04: 创建支付（未登录）**
- 操作：不携带 token 调用 `POST /payments/create`
- 预期：返回 401

### 场景 2: 支付状态查询

**TC-PAY-05: 查询支付状态**
- 前置：学员创建订单
- 操作：调用 `GET /payments/query/:orderNo`
- 预期：返回 `{ status: "pending" }`

**TC-PAY-06: 查询不存在的订单状态**
- 操作：调用 `GET /payments/query/ORD_NOT_EXIST`
- 预期：返回 404

**TC-PAY-07: 未登录查询支付状态**
- 操作：不携带 token 调用查询接口
- 预期：返回 401

### 场景 3: 手动确认支付

**TC-PAY-08: 手动确认支付（正常流程）**
- 前置：学员创建订单
- 操作：调用 `POST /payments/confirm`
- 预期：若支付宝未配置，返回 400 提示支付未完成

**TC-PAY-09: 手动确认支付（无权限）**
- 操作：学员A订单 → 学员B确认
- 预期：返回 400 "无权操作此订单"

### 场景 4: 退款流程

**TC-PAY-10: 管理端退款**
- 前置：学员创建待支付订单
- 操作：管理员调用 `POST /admin/orders/:id/refund`
- 预期：待支付订单不可退款，返回 400
- 备注：退款需要在订单支付后才可操作

**TC-PAY-11: 非管理员无权退款**
- 操作：学员 token 调用 `POST /admin/orders/:id/refund`
- 预期：返回 401 或 403

### 场景 5: 购买状态检查

**TC-PAY-12: 已购课程检查（未支付）**
- 前置：学员创建订单未支付
- 操作：调用 `GET /orders/check-purchase/:courseId`
- 预期：返回 `{ purchased: false }`

**TC-PAY-13: 已购课程检查（未购买）**
- 操作：调用 `GET /orders/check-purchase/99999`
- 预期：返回 `{ purchased: false }`

### 场景 6: 权限安全

**TC-PAY-14: 学员可访问支付创建接口**
**TC-PAY-15: 管理员不可访问支付创建接口（角色限制）**
**TC-PAY-16: 未登录无权创建支付**

## 六、已知限制

1. 测试环境支付宝未配置时，`POST /payments/create` 返回 400 是符合预期的
2. 退款完整流程（支付后退款）需要支付宝已配置场景下测试
3. 异步通知 `POST /payments/notify` 无法通过本地 API 测试（需支付宝服务器回调）
4. `GET /payments/query/:orderNo` 仅查询本地数据库状态，不包含支付宝远程查询（服务端实现仅查询本地）
