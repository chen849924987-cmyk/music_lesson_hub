# 性能优化方案

> **版本**: v1.0
> **最后更新**: 2026-05-21
> **适用范围**: 全栈（Nginx + NestJS 后端 + Vue 前端 + MinIO 对象存储）
> **优化背景**: Tailscale Funnel 外网访问模式下，页面加载和视频播放出现严重性能瓶颈

---

## 一、性能问题诊断报告

### 1.1 问题概述

| 接口/资源 | 加载耗时 | 严重程度 |
|-----------|---------|---------|
| `/api/v1/courses`（首页课程列表） | ≈4-5s | 🔴 严重 |
| 课程封面图片加载 | ≈5s | 🔴 严重 |
| 分类数据加载 | ≈4s | 🟠 中等 |
| 视频播放 | **永远加载不完** | 🔴 致命 |
| API 页面级联请求 | ≈10s+ | 🔴 严重 |

### 1.2 根因分析

#### 瓶颈 1：Tailscale Funnel 带宽瓶颈（最严重）

**根因**：Tailscale Funnel 不是 P2P 直连，通过 **DERP 中继服务器** 转发流量，中继带宽极其有限（通常 1-5 Mbps）。

**受影响资源**：
- 所有视频文件（需经过 MinIO → Nginx → Tailscale 中继 → 浏览器）
- 所有课程封面图片
- 所有 API 请求

**请求路径**：
```
浏览器 → Tailscale Funnel → DERP中继 → Tailscale节点 → Docker Nginx → 
  ├── /api/* → backend (NestJS)
  └── /minio-files/* → minio (MinIO)
```

#### 瓶颈 2：封面图片逐个生成签名 URL

**根因**：`courses.controller.ts` 中每查询一次课程列表，就对每张封面调用 `transformCoverToUrl()`，生成24小时有效期的预签名 URL。100条课程就需要 **100次 MinIO 签名请求**。

#### 瓶颈 3：视频无转码直接流式传输

**根因**：`VideosService.uploadFile()` 直接将原始视频文件上传 MinIO，不做任何转码。MP4 格式不支持真正的流式播放，浏览器需下载完整文件才能开始播放。大视频文件通过低带宽中继传输导致永远加载不完。

#### 瓶颈 4：首页 pageSize 过大

**根因**：`HomePage.vue` 中 `getPublishedCourses({ pageSize: 100 })` 一次加载100条课程数据，在 Tailscale 低带宽环境下每条数据的往返延迟累积。

#### 瓶颈 5：Nginx 代理配置未优化

**根因**：Nginx 配置未启用 keepalive 连接复用，每个 API 请求都要新建 TCP 连接，在 Tailscale 环境下每次握手增加 50-200ms 延迟。

## 二、已实施的优化（v1.0）

### 2.1 优化1：减少首页 pageSize + 分页加载

| 项目 | 内容 |
|------|------|
| **改动文件** | `frontend/src/views/dashboard/HomePage.vue` |
| **改动内容** | `pageSize: 100` → `pageSize: 12`；新增"加载更多"分页按钮（每次 12 条）；分类数据从独立 API 获取 |
| **预期效果** | API 响应从 ≈4s→**<1s**；分类和筛选标签完整显示 |
| **风险等级** | ⭐ 极低 |

### 2.2 优化2：封面启用 Nginx 直链 + 浏览器缓存

| 项目 | 内容 |
|------|------|
| **改动文件** | `frontend/nginx.conf`、`backend/src/modules/storage/storage.service.ts`、`backend/src/modules/courses/courses.service.ts` |
| **改动内容** | 封面资源通过 Nginx 代理 MinIO 直链输出，设置 `Cache-Control: public, max-age=86400`；StorageService 新增 `getPublicUrl()` 方法；`transformCoverToUrl()` 改用公开直链替代签名 URL |
| **预期效果** | 封面加载从 ≈5s→**<0.5s** |
| **风险等级** | ⭐ 低（封面变为公开，但对教育平台影响极小） |

### 2.3 优化3：Nginx keepalive 连接池

| 项目 | 内容 |
|------|------|
| **改动文件** | `frontend/nginx.conf` |
| **改动内容** | 新增 `upstream backend_upstream`（keepalive 32）和 `upstream minio_upstream`（keepalive 16）；`/api/` 和 `/minio-files/` 均启用 keepalive 连接复用 |
| **预期效果** | 多个 API 请求复用 TCP 连接，在 Tailscale 环境下整体提速 **20-30%** |
| **风险等级** | ✅ 极低 |

### 2.5 优化5：Redis 缓存加速（后端 API 数据缓存）

| 项目 | 内容 |
|------|------|
| **改动文件** | `backend/src/config/redis.config.ts`、`backend/src/common/redis/redis.service.ts`、`backend/src/common/redis/redis.module.ts`、`backend/src/modules/courses/courses.service.ts`、`backend/src/app.module.ts` |
| **改动内容** | ① 新建 RedisModule + RedisService，封装 ioredis 提供统一缓存 API（get/set/del/delByPattern/incr/getJson）；② 课程目录 `findPublished()` 无关键词搜索时缓存 5 分钟，带搜索时不缓存确保实时性；③ 写操作（编辑/审核/推荐切换）自动清除全部课程目录缓存 + 对应课程详情缓存；④ 支持优雅降级（`REDIS_ENABLED=false` 或连接失败时业务不受影响） |
| **预期效果** | 用户浏览课程目录时，相同查询条件直接从 Redis 读取（微秒级），不再重复查询 MySQL，高并发场景下数据库压力显著降低 |
| **风险等级** | ✅ 极低（有优雅降级保护） |

### 2.4 优化4：HTTP Range 请求支持（视频流式播放）

| 项目 | 内容 |
|------|------|
| **改动文件** | `frontend/nginx.conf` |
| **改动内容** | `location /minio-files/` 中新增 `proxy_set_header Range/If-Range` 转发 Range 头；添加 `add_header Accept-Ranges bytes` 通知浏览器支持分片下载；`proxy_cache_valid` 增加 `206` 状态码缓存 |
| **预期效果** | 浏览器使用 Range 分片请求视频，拖动进度条更快，首屏加载更快；**无需修改前端代码** |
| **风险等级** | ✅ 极低 |

## 三、待实施的优化方案（按优先级排序）

### 🔴 P0：视频迁移到云存储（最根本解决方案）

**方案**：将视频文件从 MinIO 迁移到阿里云 OSS / 腾讯云 COS / 七牛云等对象存储服务

**预期效果**：
- 视频加载从不完成 → 秒开
- 利用云存储的 CDN 加速
- 消除 Tailscale Funnel 带宽瓶颈

**工作量**：中（需改造 StorageService 支持多云存储适配器）

### 🟠 P1：引入 HLS 视频转码

**方案**：在视频上传后用 FFmpeg 进行 HLS 转码（`.m3u8` + `.ts` 分片）

**预期效果**：
- 真正的流式播放，首屏秒开
- 自适应码率，网络差自动降清晰度
- 浏览器只下载当前观看片段

**工作量**：高（需添加转码队列、状态管理、HLS.js 播放器适配）

**替代方案**：使用第三方视频托管平台（B站、Vimeo、阿里云视频点播）

### 🟡 P2：视频背景替换

**方案**：Hero 区域视频背景替换为 CSS 动画或小文件循环视频

**预期效果**：首屏渲染速度提升

**工作量**：低

### 🟢 P3：数据库查询索引优化

**方案**：为 `courses` 表的 `status`、`categoryId`、`teacherId` 等常用查询字段添加数据库索引

**预期效果**：大数据量下查询效率提升

**工作量**：低

## 四、技术方案详解

### 4.1 封面直链方案

**当前问题**：每张封面调用 `getPresignedUrl()` 生成临时 URL，经过 Tailscale 中继传输

**优化方案**：

1. 后端 `StorageService.getPresignedUrl()` 在 Tailscale 模式下返回 Nginx 代理路径：
   ```
   https://{TAILSCALE_HOST}/minio-files/music-edu/{objectName}
   ```
2. Nginx `/minio-files/` 配置 `proxy_buffering on;` 和 `proxy_cache_valid 200 24h;`
3. 设置响应头 `Cache-Control: public, max-age=86400` 让浏览器缓存24小时
4. 封面更新后通过修改 URL 尾部参数 `?t=时间戳` 使缓存失效

### 4.2 Nginx keepalive 方案

```nginx
upstream backend_upstream {
    server backend:3000;
    keepalive 32;
}

location /api/ {
    proxy_pass http://backend_upstream;
    proxy_http_version 1.1;
    proxy_set_header Connection "";
    # ... 其他配置
}
```

### 4.3 资源缓存策略

| 资源类型 | 缓存时间 | 缓存策略 |
|---------|---------|---------|
| HTML 文件 | 不缓存 | `no-store` |
| JS/CSS（带 hash） | 7天 | `public, immutable` |
| 课程封面 | 24小时 | `public, max-age=86400` |
| 字体文件 | 30天 | `public, immutable` |
| API 响应 | 不缓存 | — |
| 视频文件（分片） | 7天 | `public, max-age=604800` |

## 五、监控与验证

1. **浏览器 DevTools Network 面板**：对比优化前后的加载耗时

3. **`docker compose logs -f backend`**：观察后端请求处理时间
4. **Tailscale Funnel 带宽监控**：确认中继带宽使用情况
