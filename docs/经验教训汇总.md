## 一、NestJS 后端

### 1.1 路由顺序敏感（高频）

**问题**：具名路由（`@Get('pending-review')`）定义在通配路由（`@Get(':id')`）之后，通配路由会劫持具体路径导致 400 错误。

**规则**：具体路径路由**必须**定义在通配参数路由之前。具名路由上方添加注释：
```typescript
/** 必须放在 @Get(':id') 之前，否则 :id 会劫持 {path} 路径 */
@Get('pending-review')
findPendingReview() {}
```

**相关记录**：v2.7 待审核课程列表 400 错误

### 1.2 类型ORM 联合类型字段列声明（高频）

**问题**：`number | null`、`string | null` 等联合类型字段不指定 `type`，TypeORM 推断为 `Object`，MySQL 不支持导致启动崩溃。

**规则**：联合类型字段**必须**显式指定 `@Column({ type: 'int', nullable: true })`，绝对不能省略 type。

**相关记录**：v2.6 附件模块 lessonId 列类型不支持

### 1.3 手写 SQL 列名与数据库一致

**问题**：TypeORM `@Column()` 未指定 `name` 时，数据库列名 = TypeScript 属性名（驼峰）。手写原生 SQL 时用下划线列名会报 `Unknown column`。

**规则**：
- 如果 `@Column()` 没有 `name` 属性，SQL 中用驼峰列名
- `getRawMany()` 返回的 key 是 `表名_字段名` 格式（如 `course_courseType`），需手动 `map()` 转换
- `bigint` 字段在 `getRawMany()` 中返回字符串，需 `Number()` 转换

**相关记录**：v2.7 附件审核页 course_type 列名错误

### 1.4 JWT payload 禁止手动设置 iat/exp

**问题**：在 payload 中手动设置 `exp: 0` 与 `JwtModule` 配置的 `expiresIn: '7d'` 冲突，导致签名失败。

**规则**：JWT payload 中**禁止**手动设置 `iat` 和 `exp` 字段，让 `JwtModule.signOptions.expiresIn` 自动处理。

**相关记录**：v0.1.0 登录接口 500 错误

### 1.5 @CurrentUser 必须有守卫配合

**问题**：接口使用 `@CurrentUser('sub')` 但没有 `AuthGuard`，导致 `request.user` 始终为 `undefined`。

**规则**：
- 需要用户身份但允许未登录 → 使用 `OptionalJwtGuard`
- 必须登录 → 使用 `AuthGuard('jwt')`
- 全局搜索 `@CurrentUser` 用法，确保每个使用处都配对了守卫

**相关记录**：v3.3 已购课用户试看问题

### 1.6 ApiResponse.successWithPagination() 不保留额外字段

**问题**：该方法只接受 `items: T[]` 和 `meta: PaginationMeta`，会丢弃响应对象中的其他字段（如 `stats`）。

**规则**：需要在分页响应中返回额外数据时，必须使用 `ApiResponse.success({ items, meta, stats })` 包装自定义对象。

**相关记录**：v3.1 超管订单管理统计数据丢失

### 1.7 Repository.manager 优先于 DataSource 注入

**问题**：在 Service 中注入 `DataSource` 获取 `EntityManager`，导致模块间不必要的依赖。

**规则**：使用 `this.repository.manager`（每个 repository 都自带 `manager` 属性）：
```typescript
// ✅ 推荐：直接通过 repository 获取
const manager = this.userRepository.manager;
```

**相关记录**：v2.7 附件审核页 500 错误

### 1.8 DTO 查询参数类型转换（高频）

**问题**：`@Query()` 接收的值默认为字符串，`@IsInt()` 校验字符串 `"12"` 失败。

**规则**：在 DTO 字段上添加 `@Type(() => Number)` 配合 `@IsInt()`：
```typescript
@IsInt()
@Type(() => Number)  // 查询参数字符串 → number 自动转换
categoryId: number;
```
这同样适用于 `multipart/form-data` 中的数字字段。

**相关记录**：v0.2.0 分类查询参数类型转换、v2.6 附件上传 DTO

### 1.9 状态校验更新需闭环（v2.4 后新增）

**问题**：新增状态流转路径时容易忘记同步更新验证逻辑。

**规则**：修改课程状态流转时，同时检查：
1. `validateStatusTransition()` 状态表
2. `submitForReview()` 允许状态列表
3. 相关业务方法中的 `if` 判断

**相关记录**：v2.5 已驳回课程重新提交、v2.6 OFF_SHELF→PENDING

## 二、Vue 前端

### 2.1 Pinia Store 响应式解构

**问题**：`const isDark = themeStore.isDark` 是一次性赋值，store 变化时不会同步更新。

**规则**：
- 模板中直接使用 `store.xxx` 是最安全的做法
- 如果需要在 script 中解构，必须使用 `storeToRefs(store)`：
```typescript
const { isDark } = storeToRefs(themeStore);
```

**相关记录**：2026-05-16 ThemeToggle 图标不刷新

### 2.2 scrollBehavior 不适用于懒加载组件

**问题**：路由层 `scrollBehavior` 执行时，懒加载组件尚未挂载，DOM 元素不存在，滚动必然失败。

**规则**：
- 跨组件 hash 跳转：放弃路由层处理，改由目标组件在 `onMounted` 中主动检查并执行滚动
- 同一组件内路径切换：可在路由层用 `scrollBehavior` 处理
```typescript
onMounted(() => {
  nextTick(() => {
    const hash = window.location.hash;
    if (hash) {
      document.querySelector(hash)?.scrollIntoView({ behavior: 'smooth' });
    }
  });
});
```

**相关记录**：2026-05-16 锚点跳转不生效

### 2.3 Vue 应用避免原生 `<a>` 做内部导航

**问题**：原生 `<a href="/home#about">` 会导致页面刷新，丢失 SPA 状态。

**规则**：Vue 应用中所有内部导航必须使用 `<router-link>` 或编程式导航（`router.push()`）。

**相关记录**：2026-05-16 锚点跳转不生效

### 2.4 路由路径必须精确匹配注册路径

**问题**：
- 独立全屏路由（`/checkout`）和嵌套子路由（`/producer/cart`）是不同路由
- 跳转时必须使用注册时定义的完整路径

**规则**：
- 推荐使用 `router.push({ name: 'RouteName' })` 命名路由跳转
- 使用路径字符串时需确认路由注册路径（检查 `router/index.ts`）
- 404 时浏览器标题会显示 "404 - SoundCraft"，是排查信号

**相关记录**：v3.1 结算页/详情页跳转 404

### 2.5 vue-tsc 未使用变量处理

**问题**：模板 ref 变量被模板使用但 TypeScript 类型检查认为未使用。

**规则**：在变量声明后添加 `void` 标记显式告知编译器：
```typescript
const heroVideoRef = ref<HTMLElement | null>(null);
void heroVideoRef; // 告知 TS 已被模板使用
```

**相关记录**：v2.7 vue-tsc 构建警告

### 2.6 Vue SFC 文件末尾检查

**问题**：`write_to_file` 工具可能将 XML 标签写入文件末尾，导致 Vue 编译错误。

**规则**：Vue SFC 文件必须以 `</style>` 结尾，之后没有任何额外字符或标签。创建新页面后立即检查文件末尾。

**相关记录**：v3.1 MyOrders.vue XML 标签残留

### 2.7 浮点数运算结果必须取整

**问题**：倒计时显示出现小数秒（`4:59.8`）。

**规则**：UI 展示前用 `Math.floor()` 或 `Math.round()` 取整，然后做取模运算：
```javascript
const totalSecs = Math.floor(remaining);
const secs = totalSecs % 60; // 先取整再取模
```

**相关记录**：v3.3 试看倒计时小数

## 三、价格与金额处理

### 3.1 价格分↔元转换规范

**标准链路**：
| 节点 | 操作 | 单位 |
|------|------|:----:|
| 前端输入 | — | 元 |
| DTO 接收 | — | 元 |
| Service.create/update | ×100 | 分 |
| 数据库存储 | — | 分 |
| Controller 返回 | ÷100 | 元 |
| 前端展示 | — | 元 |

**关键规则**：
- Service 层做"元→分"（输入转换）
- Controller 层做"分→元"（输出转换）
- **所有价格字段统一处理**：`price`、`originalPrice`、`singlePrice` 等
- 前端不应感知存储单位，不出现 `/ 100` 代码

**相关记录**：v3.3 课程价格少零、v3.4 Checkout 金额重复除以100

### 3.2 后端价格转换策略统一

**问题**：课程价格做了分→元转换，但课时单独价格（`singlePrice`）没做，前端对不同源做不同处理。

**规则**：后端所有 Controller 返回的价格字段都必须统一转换，建议在 Controller 或拦截器层集中处理。

**相关记录**：v3.4 Confirm 订单页金额重复除 100

## 四、支付宝支付

### 4.1 alipay-sdk v4.x exec() 返回值结构

**规则**：`alipay-sdk v4.x` 的 `exec()` 直接返回 camelCase 结果对象，而非 `result.alipay_trade_query_response` 嵌套结构：
```typescript
// ❌ 旧写法
const response = result.alipay_trade_query_response;
// ✅ 新写法（v4.x）
const tradeStatus = result.tradeStatus;
const subCode = result.subCode;
const tradeNo = result.tradeNo;
```

**相关记录**：v3.2 支付确认接口 Bug

### 4.2 密钥三件套

**规则**：支付宝支付需要三组密钥，任何一组不匹配都会导致功能异常：
1. **应用私钥**（PKCS8）：后端签名用
2. **应用公钥**：上传到支付宝开放平台
3. **支付宝公钥**：从支付宝开放平台获取，用于验签

应用公钥 ≠ 支付宝公钥，两者完全不同，不能混用。

**相关记录**：v3.2 支付宝公钥不匹配

### 4.3 .env 变更后需重启后端

**规则**：修改 `.env` 中的配置后必须重启 NestJS 后端才能生效（配置在构造函数中只加载一次）。

## 五、其他

### 5.1 前端不应感知存储单位

**原则**：数据库以"分"为存储单位（支付标准），但前端所有页面看到的应该是"元"，后端负责所有单位转换。这样即使未来更换存储单位，前端也无需改动。

### 5.2 Toggle 开关应独立保存

**原则**：Toggle 开关、Checkbox 等即时切换控件应该 `onChange` 立即调用 API 保存，不应等待用户点击独立的保存按钮。保存失败需自动回滚至之前状态。

**相关记录**：v3.1 通知偏好耦合

### 5.3 Controller 层不截断额外字段

**原则**：Service 层返回完整数据 → Controller 层包装时需要确保不截断字段，尤其是使用泛型方法（如 `successWithPagination()`）时。如需保留额外字段需改用 `success()`。

**相关记录**：v3.1 超管订单管理统计数据丢失

### 5.4 种子数据应与产品设计一致

**原则**：种子数据不仅是测试数据，更是开发人员理解系统设计的参考，必须符合实际业务规则（如单课=1个课时）。

**相关记录**：v3.3 单课种子数据问题

### 5.5 各端接口权限分层

**原则**：各端调用接口应该清晰区分：
- `GET /courses` → 公开（已上架课程浏览）
- `GET /courses/teacher` → 教师端（教师自己创建的课程）
- `GET /courses/admin` → 管理端（全量课程管控）
- `GET /courses/producer` → 制作人端（已购买的课程，待开发）

**相关记录**：v2.7 制作人端 403
